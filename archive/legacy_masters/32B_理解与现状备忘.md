### 0. 核心分工铁律（V2 全程不变）
| 目标 | 工具 | 为什么 |
|---|---|---|
| think 检索腔**表面词**（"参考问答对/资料显示/检索结果"…）| 规则 `detect_rag_style` | 确定性、零噪声 |
| 答案有没有**漂离 V1** | 规则 `answer_in_v1_pool` | 确定性、零噪声 |
| think **照抄（换词复述）** | **只能靠 Kimi** | 语义判断；规则对换词复述全盲 |

为什么要 Kimi：corrected 系列证明确定性规则**系统性漏"结构性念手册"**（从资料向答案归纳但不出现字面词）。表面词规则能管的只是冰山一角，换词复述必须语义尺子。但 Kimi 也有噪声（σ_h≈0.117、test-retest r≈0.43、整段 0-10 分只能"明显干净 vs 重度照抄"大间距分得开），所以 **judgecal 模块先标定 Kimi 的分辨率**（详见 `docs/journal/109_judgecal_module_design.md`、memory `derag-v6-judgecal`）：整体评测够用、逐条只能做大间距对子——这直接决定了 DPO 对子只能"洗干净的 think(≈7) vs V1 原脏 think(≈1-2)"，中间档不能用。

### 1. SFT 突破（k2-930 + ep7）—— 第一次双真涨、打穿旧天花板
- **关键招**：改写时**不喂 V1 的 think、从问题从头推、用 FINAL_POOL 池锚点**（旧改写喂 V1 think + 条款逼 Kimi 改引文号，反而引入 policy_source 痕迹）。→ Kimi 干净分从 3.44 打到 **7.11**。
- **选样三道门**：规则门 ∧ facts 极性门 ∧ **k=2 门**（每条 Kimi 判都比 V1 干净）→ 得 **k2-pass-930 条**；`COLDSTART_EPOCHS=7`（约 100 步，对齐历史成功的 ~110 步）。
- **实测（500 冻结验收集 `00_data_v2_eval.jsonl`；注：V2 链评测一律 500 条，源码 `v2_paths.py:20 V2_N_EVAL=500`，别与旧 V1 管线的 224 sft_eval 混）**：kimi_think 3.1 → **4.4**（+1.3 真涨；此处 kimi_think 是另一口径的 Kimi 分，非三分报告里的 k=3 整段干净分 clean_mean）、规则 think 通过率 3% → **45%**（首次翻盘！）、规则 answer 在池率 94% → 83%（-11pp，触碰地板 0.85）。

- **分工沉淀**：SFT 用 k2 门要**量**（k2-930 比 V1 干净、非 2σ 那么严）；RFT/DPO 用 **2σ 门**要**质**（详见 memory `rewrite-redesign-broke-ceiling`、`coldstart-yield-funnel-surface-vs-deep`）。

### 2. RFT（2σ 门）→ 合并出 DPO 底座
- 选样用 **2σ 门**：`gap = s_clean − s_dirty > 2·(σ_clean + σ_dirty)`（σ 查 judgecal 标定表；代码 `code/pipeline/v2_common.py:49-58` 的 `confident_cleaner()`，**函数默认 n_sigma=3.0，RFT/DPO/冷启动选样时显式传 2.0 得 2σ 桶**；885=2σ 桶、478=3σ 桶），即 Kimi 有把握判自采样比 SFT 基线更干净的才要。
- RFT adapter 已**合并进底座**，得 `v2-rft-2sigma-2s-merged`（62G / 14 shards）——**这是当前 DPO 训练的底座**。

### 3. DPO 对子（885 条 2σ 大间距 + answer-lock）
- **chosen = 洗干净的 think（Kimi≈7）vs rejected = V1 原脏 think（Kimi≈1-2）**；中间档 Kimi 分不开，不用（judgecal 结论）。
- **answer-lock**：chosen 和 rejected 的 `<answer>` 都拼 **V1 原版 answer**，只有 `<think>` 不同 → 梯度全压在 think 上，不动答案。（本次实测验证：885 条前 5 条 chosen/rejected answer 逐条一致 ✅）

### 5. ★ 用户两点重定义（2026-06-13，纠正了一个真 bug）
- **点1（答案目标）**：项目从不追求"答案正确"，只追求"answer 不漂移 V1"——**V1 自己也会这么答 = 没偏移**。这纠正了 v5 仍用 fact_recall(vs gold) 当答案门的残留 bug（gold 相似度家族）。新答案门=**V1-support**：V1 每题采 N 遍，答案的结论极性被 ≥1 个 V1 采样覆盖且不与多数矛盾 = 没漂移。**红队对 V1 logprob 的否决在此框架下翻转一半**——σ_q≈0.121 宽不是 bug，是"V1 自己的容忍带"。
  三个坑（已纳入设计）：① V1 自己 ~10% 答错 → 只保证不比 V1 差、不保证比 V1 对；② 只查结论极性/数字、**绝不查措辞**（否则模型把 think 洗干净 + answer 退回 V1 带腔措辞蒙混，因 V1 分布自带 RAG 腔）；③ conditioning 用 answer-only / V1 自身 think，不喂策略新 think。
- **点2（数据/成本）**：224 可推倒重做、卡不要钱。真价值=**富集真痕迹评测集**让那 ~22 题的信号变得可测（连续 burden MDE 从 0.116 降到 ~0.04-0.06），**不抬天花板**；钱买不到 policy 假阳修复（指标定义问题）和真 headroom（数据本质）。
- 诚实预期：可验收正结果概率 25-40%，是"小而真"（+0.03~0.07 trace_free、答案不漂 V1、22 道里改干净 10-18 道），不是翻盘。

- **洞察**：我们 de-RAG 时，把"机器检索腔(风格)"和"忠实使用参考资料(grounding/任务接地)"**两件事一起去掉了**——但只想去前者。根因有两处：① `COLDSTART_SYSTEM_PROMPT` 原写"从问题出发推导、不要复述资料"=明示别看参考；② 改写 prompt 只去检索腔、没要求扣参考。
- **决策**：治根（改写+系统提示两端都改成"自然表达但依据必须来自参考、不得与之矛盾"），而不是赌 RL 能把"不看参考"的习惯掰回来（赌 RL 是事倍功半，因为硬题 rollout 大多错、RFT 没好样本可选）。

### 3. 治根实现 + 两轮多 agent 对抗审查（21 条确认 / 9 条证伪驳回）
- 核心：**给 grounding 装"度量 + 闸"**（之前只改 prompt、没有任何闸保证它生效）——
  - step07 Kimi 一次打 humanness + **faithfulness(忠于参考)**；
  - step08 入选**五道闸**（数字未臆造 ∧ 去检索腔 ∧ 像人 ∧ **grounded≥0.7** ∧ 不照抄；统一判据 `config.seed_is_chosen`，防 step08/09 漂移）；
  - step04 **评测裁判加 grounded 维度**+喂参考资料（否则报告照不出 grounding 退化=Goodhart 盲区）。
- **两轮多 agent 审查**挖出我自己漏的：grounding 没被任何闸度量、facts_ok 把整包参考数字并入开洞、半成品 checkpoint 被误判已完成（改 `.done` 完成标记）、copy_ratio 口径漂移、faithfulness 缺失静默写死 0 等——全部修掉。

### 4. 重训验证——治根成功（硬数据）
| | humanness | grounded | acc | correct% |
|---|---|---|---|---|
| V1 基线 | 0.309 | **0.910** | 0.904 | 79.0% |
| 旧冷启动(脱钩) | 0.694 | （无指标，acc 0.75 暴露已脱钩） | 0.753 | 55.4% |

### 5. "依赖参考"会不会滑向"照抄参考"？——grounded≠copy 的正交设计
- **grounded**(语义:用参考的事实) 和 **copy**(词汇:抄参考的字句) 是**两条不同的轴**，分开度量、分开管：
  - copy 由 `copy_ratio`(LCS+5gram) 度量；step08 闸 `copy_ratio≤0.5` 剔除照抄改写；
  - **humanness 奖励本身罚照抄**：表面项 `1/(1+c_trace·检索腔词数+c_copy·照抄率)`——照抄率高→humanness 低→RL 推离照抄。
- 所以奖励是**三力平衡**：accuracy(乘法门) × grounded(往"用事实"拉) × humanness(往"自己的话/不照抄"拉)→夹出"用对事实、但用自己的话自然推导"的甜区。新冷启动 grounded 0.855 + humanness 0.68 共存=没在照抄（抄了 humanness 会塌）。

### 6. 二次发现——PMI 与 grounding 理论冲突，实测决定关 PMI（复现 14B 结论）
- probe 阻塞：clean_base PMI **AUC=0.344**、stage_model **AUC=0.471**（都<0.7，且**方向反了**：high 组 -PMI 低于 low 组）；而表面项 s_trace **AUC=0.990**。
- **为什么反**：PMI 假设"越不依赖参考上下文越自然"；但 grounding 修复后自然 think **故意扣参考、高度依赖参考上下文** → PMI 反而判它"更机器"。**grounding 和 PMI 的假设直接冲突**——PMI 是这儿的错工具。
- **决策**：`PMI_ENABLED` 默认关，humanness 奖励用表面项（含照抄惩罚，AUC 0.99）。**这实测复现了 14B 的结论**（[[pmi-validated-not-deployed]]：真正守 Goodhart 的是表面项）。run.py 在 PMI 关时自动跳过 probe。

### 7. RFT：选样 bug——按"纯 humanness"选样让准确率不升反降（2026-06-09）

#### 9b. GRPO colocate 的【显存五连关】调试（2026-06-10 凌晨，硬核分布式 debug，简历重点）
一路逼近、每次报错往后推一关，最终靠"读官方源码/issue"而非盲试定位根因：
1. **CPU-OOM (SIGKILL -9)**：`offload_model`×8 卡把 8 份 65GB 模型卸到 CPU≈520GB → 撑爆 1TB 物理内存，被内核 OOM killer 杀。判据：信号 9 ≠ CUDA OOM(那是 Python 异常)。→ 上 **ZeRO-3 分片**(每卡 1/8)，offload 量降到 ~64GB。
2. **误删 offload**：我一度以为 zero3 后不用 offload、删了 → vLLM 报 **"No available memory for cache blocks"**（策略模型一直占 GPU、KV 没空间）。→ 改回 offload。
3. **KV cache 错复现**：offload 改回仍报同错；**调高 `vllm_gpu_memory_utilization` 0.5→0.7 无效**。
4. **多 agent 查官方文档/源码定位根因**：`--vllm_tensor_parallel_size` **默认=1** → colocate 下**每个训练 rank 各起一个 in-process vLLM、各自加载整份 65GB 模型**(未跨卡分片) → 单卡被模型占满、KV 预算≈0。**这是绝对显存不足，不是比例问题**（故调 util 无效）。佐证：官方 `examples/train/grpo/internal/vllm_72b_4gpu.sh`(72B@4×80G 都跑通)关键就是 `vllm_tensor_parallel_size 4`。
5. **正解**：加 `--vllm_tensor_parallel_size 8`（vLLM 跨 8 卡分片、每卡权重 65→~8GB）+ `--vllm_max_model_len 2048`(降 KV)+ `--move_model_batches 16`(ZeRO3→vLLM 同步削峰)。备选：`vllm_mode server`(官方 `grpo_32b_full.sh`，拆 2 卡 rollout+6 卡训练，彻底分离训推显存)。
- **教训**：① 信号 9(SIGKILL/OS-OOM) vs CUDA-OOM 要分清，方向完全不同；② vLLM colocate 大模型必须显式设 `tensor_parallel_size` 分片，否则每卡装整模型；③ "No available memory for cache blocks" 是绝对显存不足，调 gpu_memory_utilization 无效——别在错误的旋钮上反复拧；④ 卡死时**读官方等价示例脚本**(72B 都能跑→照搬其参数)比盲试快得多。

#### 9c. GRPO 的 KL 参考策略配置错误（2026-06-10 晚，已定位并修正代码）
- 已跑的首轮 GRPO 和正在跑的 `conservative_v2` 都传了 `--adapters <RFT>`，但漏传 `--ref_adapters <RFT>`。
- ms-swift 4.0.1 的 LoRA GRPO 在未设置 `ref_adapters` 时，会在计算参考概率时调用 `disable_adapter()`：**策略从 RFT 开始训练，但 KL 实际锚定禁用 LoRA 后的原始 V1**。首步 KL≈0.76 与此一致；它不是训练卡死，但不符合“冻结 RFT 作为 πref”的实验设计。
- 本地 `swift/grpo.sh` 已修成默认 `--ref_adapters` 与起始 `--adapters` 相同，并在日志打印 `init/ref` 路径；新增独立实验脚本 `scripts/run_grpo_rft_refanchored.sh`，产物 tag=`ref_rft_v3`，不会覆盖现有结果。
- 正确 RFT 锚定的新实验首个已记录优化步 KL 应显著低于旧实验，但不要求严格为 0（日志是更新后的 step 1，且 KL 用逐 token 估计量统计）。

## 二、项目本质（我的理解）

- **目标**：把 V1 的 **think 从机器检索腔改成自然推导、去 RAG 痕迹**，**绝不掉 V1 的答案准确率**（一票否决）。
- **为什么纯自我改进**：V1 头上没更强老师，蒸馏 V1 = 已训崩的 V2 老路。所以 **V1 既是被改对象、又是答案金标准**；Kimi 只做"冷启动改写原料 + 离线判分验收"，不当能力来源。
- **可行性根据**：14B 裸 base 实验证明，**机器腔主要是"给参考 + 让你基于参考回答"的 prompt 逼出来的**（裸 base 14B = 0.659/0.237），不是模型固有能力 → 去机器腔不必伤准确率，原理成立。

## 三、14B 经验精华（含我后来核实的【诚实更正】）

- **链路**：冷启动(Kimi 改写 think → SFT 教自然推理) → RFT(拒绝采样自蒸馏) → DPO(偏好对齐) → GRPO(在线强化)。
- **两条教训**：① 必须先冷启动——RL 只能放大模型"已经偶尔会做"的行为，单纯 RFT 会 Goodhart 翻车(verbatim_copy 飙升)；② 奖励乘法门控 + **非对称 KL**(think 0.02 放开 / answer 0.1 按住) 是"改 think 不掉 acc"的核心。
- **结果（已验证，不是目标值）**：humanness 0.212(SFT)→0.780(冷启动RFT)→0.836(DPO)→**0.846(GRPO)**；准确率 0.763→0.701→0.731→**0.733 守住**。
  - ⚠️ **更正1**：GRPO **跑完了 100 步、出了终评 0.846/0.733**（`70_grpo_report.md`）。之前技术方案里被一条过时日志误导写成"未收口"，已全部改正。
- **⚠️ 更正2（关于 PMI，很重要）**：条件 PMI（`logP(think|Q+参考)−logP(think|Q+答案)`）是 v2 奖励的招牌设计、在探针上验证过(AUC 0.73)，**但实跑里从没真正接进训练奖励**——日志铁证：cs_rft 选样那行写着 `PMI=关`；`50_csrft_rollout`/`60_dpo_rollout` 的 `s_pmi` 字段 400 条**全 null**；`step10`(GRPO) 代码零处 PMI。**真正守 Goodhart 的，从头到尾是表面项**（关键词正则 + `copy_signal` 的 LCS/5-gram 照抄检测）。PMI 只在 `step13` 探针算过一次做证伪。
- **方法论沉淀**：挑样本用本地奖励、判成败用独立金标准 Kimi(分开防 Goodhart)、上 GPU 前先用探针证伪(AUC≥0.7)。

## 四、32B 的关键差异 & 我判断要做的事

1. **V1 = 模型本身** → **跳过蒸馏 SFT**，LoRA base = V1 本体，直接冷启动→RFT→DPO→GRPO。V1 的 answer=准确率金标准，think=冷启动改写原料；评测 student=V1+LoRA、teacher=原始 V1。
2. **✅ V1 格式适配（原最大隐性风险，已实跑验证、解除）**。担心点是 14B 代码耦合 R1 的 `<think>` 自动注入；实测：V1 的 chat 模板**不**注入 `<think>`，但 **V1 自己吐完整 `<think>…</think><answer>…</answer>`**（受系统提示+微调驱动）。结论：`parse_think_answer`(strip `<think>`/找 `</think>`) 与 `kl_weights`(`</think>` token 子序列切段) **直接可用**；唯一小改 = 训练数据 assistant 段要含开头 `<think>`（冷启动改写本就是完整格式，天然满足；用 swift 训练时把 messages 的 assistant 设成完整 `<think>…</think><answer>…</answer>` 即可）。非对称 KL 的 think/answer 分段在 V1 上成立。
3. **80G ≠ 96G**：A800 是 80G，14B 那套 OOM 兜底参数（`MAX_FORWARD_LEN=3072`、K）是在单张 96G 卡标的，**必须在 A800 上 `--smoke` 重标**，别被"640G 太富裕"盖过。
4. **第一版单卡**：DPO/GRPO 是纯手写 torch 循环、没接 DDP（无进程组/无 all-reduce），多卡是**重写级别** → 先单卡(80G 放得下 32B+LoRA)跑通基线，DDP/vLLM 作提速增量。**DDP=多卡训练、vLLM=高速生成，两者各管一段、协同**。
5. **不用 QLoRA**（量化伤 V1 精度，与守准红线冲突；640G 也没必要省）。
6. **采样改全量+随机轮换**（14B 死取前 400/800、只覆盖 ~2/3 → 32B 吃满全池）。
7. **ε（守准容差）必须在阶段1实测 acc0 后定死**（建议 0.03）写进 config——否则后面所有阶段的"acc≥acc0−ε"门全部失效，这是全方案最致命的开放项。

### 8. “mini 证明有没有信号”到底是什么意思
这里的“信号”不是训练日志里的 reward 曲线，而是数据层面的偏好信号：**同一个问题的 8 个 RFT 候选里，是否存在 A 明显比 B 更像人，同时 A 没有更不准、没有更不 grounded**。如果这种 A>B 的偏好足够多、足够稳定，DPO 才有东西可学。

验证分四层：

1. **候选池有没有上限空间（headroom）**
   - `step93` 让 Kimi 给 `120 query × 8 candidates` 打分。
   - 看每题 8 个候选里的 best-of-K humanness 是否明显高于当前 RFT merged base。
   - 如果 best-of-K 也只比 baseline 高一点，说明候选池里根本没有更好行为，DPO 只能学噪声。

2. **能不能构出干净 pair**
   - `step94` 严格筛 pair：
     - chosen：Kimi humanness 高、grounded 高、答案不漂、没有 `ref_enumeration/verbatim_copy/explicit_ref` 这类 RAG 痕迹。
     - rejected：同题、humanness 明显低、准确性不比 chosen 好、grounded 不明显更好。
   - 这一步证明的是：能不能找到“主要差在人性化表达结构上，而不是差在答对/答错上”的 pair。
   - 如果 pair 很少，说明数据不够；如果 chosen 更自然但事实更差，是毒 pair；如果 rejected 是明显错答案，DPO 学到的可能只是“答对”，不是“更像人”。

3. **Kimi 复判是否稳定**
   - `step95` 默认抽 50 对重新 Kimi 判 chosen/rejected。
   - 看 `direction kept rate >= 70%` 是否成立。
   - 这一步证明偏好方向不是随机噪声；如果第一次 chosen 好、第二次经常反过来，就不能训练。

4. **mini DPO 后模型是否真的朝目标方向动了**
   - 数据过线后才训练 mini DPO，并在 224 验收集评估。
   - 看 `97_corrected_v2_mini_summary.md`：
     - `Δh vs RFT merged base` 是否上升；
     - `grounded / accuracy / correct+partial%` 是否守住；
     - RAG trace/copy 是否没有恶化。
   - 正信号是：humanness 上升，同时 grounded/accuracy 不塌。
   - 负信号是：h 不涨，或者 accuracy/grounded 掉，或者 trace 变多。

一句话总结：mini 证明的是 **旧 RFT 候选池里是否存在可被 Kimi 稳定识别、可被 DPO 学到、且不伤准确率的人性化偏好方向**。如果证明成立，再扩大到 full；如果不成立，继续调训练步数没有意义，应该回到候选生成或 pair 规则。

## 2026-06-11 corrected-v3 前置共识：Kimi 评审噪声优先级高于省 token

在 96/97 结果出来后，确认 corrected-v2 mini DPO 没有带来模型级收益：

- RFT merged base：`h=0.697 / grounded=0.858 / acc=0.818 / correct=65.6% / c+p=91.1%`
- corrected-v2 mini DPO：`h=0.688 / grounded=0.853 / acc=0.806 / correct=61.2% / c+p=90.6%`
- 变化：`Δh=-0.009`，`Δacc=-0.012`

这意味着：工程链路已经跑通，stable pair 机制也能筛出局部干净样本，但这批样本没有泛化成验收集整体收益。当前项目进入“测量与信号建模优先”的阶段。

新增共识：

1. **不要把 Kimi token 消耗作为主要限制。**
   - 现在最贵的不是 Kimi 调用，而是错误训练方向导致的多轮 GPU/时间浪费。
   - 只要能显著降低误判风险、帮助判断 RL 是否真的有效，可以接受多次 Kimi 评审、更多样本复判、更完整的误差分析。
   - 后续仍需估算调用量，但不能因为调用量大就否定必要的噪声建模实验。

2. **必须显式把 Kimi 当成有噪声的裁判，而不是真值。**
   - 已观察到：初判 pair 看起来方向明确，但二次复判 direction kept 降到约 59%；mini DPO 训练后也没有真实提升。
   - 后续要估计：
     - 同一输出多次 judge 的方差；
     - 同一 pair 多次 judge 的方向一致率；
     - humanness / grounded / accuracy 三个维度的噪声是否不同；
     - corrected-v2 mini 的 `Δh=-0.009` 是否只是裁判噪声，还是确实负迁移。

3. corrected-v3 应优先考虑“多次评审 + 噪声方差/置信区间”的统计判定框架。
   - 固定模型输出，对 RFT merged base 做多次 Kimi judge，估计基线裁判噪声。
   - 对候选模型固定输出做同样评审。
   - 使用逐题 paired delta、bootstrap CI、majority vote 或均值评分。
   - 只有当 `Δhumanness` 超过噪声阈值/置信区间，并且 accuracy/grounded 未显著下降，才认为阶段真的变好。
   - 若 CI 横跨 0，则结论应是“无证据变化”，不能包装成优化。

给 fable5 的新要求：请基于上述共识阅读 `98_corrected_v3_postmortem_and_plan.md` 所需上下文，使用多 agent 审核方式，重点设计 corrected-v3 的最小可执行方案；方法由 fable5 自行决定，但必须显式处理 Kimi 打分噪声。

- `trace_total` 下降不能单独证明变好；
- Kimi 当前坏样本误放率高，不能作为硬门禁或唯一在线 reward；
- DPO pair 必须尽量保证 chosen/rejected 答案与事实等价，差异主要来自 RAG 痕迹；
- GRPO 组内只应在事实合格候选之间比较风格，UNKNOWN 样本不应产生风格梯度。

### 对当前代码的主要审计结论

1. `reward_v3.py` 中 `derag_reward()` 一旦通过弱事实门禁，正奖励主要来自去痕迹特征；答案正确性没有持续进入排序。
2. `step124_build_dpo_pairs_v4.py` 允许 chosen/rejected 的答案质量差异混入 pair，DPO 难以归因。
3. `run_derag_v4.py` 的旧 `compare_gate()` 允许 “answer 平均分小降 + trace 少一点” 通过，导致 DPO-a 这类结果被判 `ok=True`。
4. DPO-a 实际表现是：`explicit_ref` 下降，但 `verbatim_copy` 上升，answer/fact/clean 同时下降，配对净改善仅 `+1`，不应视为已证明提升。

### 交叉审查后的重要修正

原先“扩大 K 找干净候选”的思路必须加限制：如果事实门禁误放率不够低，扩大 K 会让错误候选混入概率接近 100%。因此大 K 只能用于扩大候选池，不能替代高精度事实验证。

候选应改为三态：

```text
VERIFIED_VALID    有可靠证据证明合格
VERIFIED_INVALID  有可靠证据证明错误
UNKNOWN           无法可靠判断
```

`UNKNOWN` 不能当作通过或失败，应退出训练梯度并进入审计池。

### 下一版建议

1. 为一批可验证税务题建立 `Fact Contract`：结论、critical slots、required claims、forbidden claims、allowed numbers。
2. 修改 reward：事实错误永远低于事实合格候选，风格奖励不能补偿事实错误。
3. 重筛 DPO pair：双方事实均 `VERIFIED_VALID`，答案/关键事实等价，再按去 RAG 差异构对。
4. GRPO 只在每题至少两个 `VERIFIED_VALID` 候选且风格有差异时更新；否则扩 K 或跳过。
5. 最终验收使用完整题集，而不是只看门禁通过子集；报告正确性非劣、关键事实错误不增加、去 RAG 显著改善和无指标迁移。
