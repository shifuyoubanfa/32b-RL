## 〇、一页看懂（摘要）

**目标。** 公司 V1 答案准，但它的推理过程（think）满是"根据参考问答对1""查阅检索结果"这类 RAG 机器检索腔；对外宣称端到端，却在 think 里暴露了检索痕迹。本项目用**纯自我改进强化学习**，把 V1 的 think 改造得"像人、去检索腔"，并且**绝不掉 V1 已有的答案准确率**。

**为什么必须纯自我改进。** V1 头上没有比它更强的老师；蒸馏 V1 正是已经训崩的 V2 老路。因此本方案不引入任何外部老师：V1 既是被改造的对象，又是答案的金标准。Kimi 仅用于"冷启动时把机器腔 think 改写成自然推导"和"离线判分验收"，不作为能力来源。

**技术路线（一句话）。** 直接在 V1 本体上挂 LoRA，走 **冷启动(CS) → 冷启动 RFT(CS_RFT) → DPO → GRPO** 四级链路。这是 14B 沙盒已跑通经验的同构放大，关键改造点是：**跳过蒸馏 SFT**（LoRA 的 base 就是 V1 本体）、采样从"死取前 400 题"升级为**全量 + 随机轮换**、**训练改用 ms-swift 框架**（不再把 14B 单卡手写循环改造成多卡；自研 `reward.py` 与选样逻辑移植成 swift 的自定义 reward 插件，见 §4.2）、不用 QLoRA（量化会伤 V1 精度）。

**关键判据（项目成功 = 四条同时满足）。**
1. **像人**：固定留出集上，独立金标准 Kimi 判定的 think humanness 较 V1 显著提升（终评硬线 `humanness ≥ 0.70 且 ≥ 3×humanness0`）。
2. **守准（一票否决）**：student 答案准确率 `acc ≥ acc0 − ε`，ε 在阶段1实测 acc0 后立即定死（建议 ε=0.03）并写进 config，全程不破。
3. **纯自我改进**：链路无任何"比 V1 更强的老师"被蒸馏进模型；RFT/DPO/GRPO 信号均来自模型自采样。
4. **可复现**：固定种子/数据划分/冻结 checkpoint，不依赖内网（本地 V1 推理 + 公网 DashScope Kimi），组长可照文档重跑出同档结果。

**两套裁判的纪律（全程贯穿，防 Goodhart）。** 挑训练样本用**本地奖励**（快、高频、可在线）；判阶段成败用**独立金标准 Kimi**（在固定留出集上，与选样器物理分开）。每次上 GPU 前先用**信号探针廉价证伪**（AUC ≥ 0.7），先证伪再烧卡。

> **三条诚实声明（与代码核对后必须讲清，不夸大）。**
> - **条件 PMI 实际上几乎没进训练奖励——必须说清（已逐文件 grep 核对）。** (a) `pmi_cond` 算 PMI 需加载 base 模型、每候选 2 次前向，很贵；(b) 在 `step08b`(RFT 选样) 与 `step13`(探针) 都是 `--with-pmi` **可选开关、默认关**；(c) `step10`(GRPO) 与 `step08`(rollout) **零处 PMI**（`score_rollout` 不传 `s_pmi`）。**14B 实跑里 PMI 只在探针(step13)算过一次做证伪（那个 AUC≈0.73）；按 `run.py` 默认，连 RFT 选样都没开 PMI。** 即：真正喂进训练的奖励，从头到尾基本是**表面项**（关键词正则 + `copy_signal` 的 LCS/5-gram 照抄检测）在守 Goodhart；PMI 是"验证过、但没真正部署进训练循环"的信号。**因此 32B 已明确决策（本轮拍板，见 §3.2/§7-阶段3/4/6）：这次把 PMI 真接进训练**——资源够（8×80G），离线（探针 + RFT 选样）与在线（GRPO）都开 PMI。**关键澄清两点**：① **PMI 的"尺子模型"由探针 AUC 在 c(高质改写) vs 机器腔(低) 上选优**（候选：当前模型 / 冻结 πref / 干净 Qwen2.5-32B-Instruct 底座），**排除 RAG 腔的原始 V1**（拿它当尺子最差）；② **PMI 里的模型是"度量尺子"、不是被奖励的对象**——奖励的始终是当前模型生成的 think，PMI 只是和正则/LCS 并列的一把"用语言模型概率做的尺子"。
> - **本地奖励的"准确率门"是"答案漂移门"，不是"答对率"。** 代码核对：`answer_drift()` 算的是生成 answer 与 V1 answer 的相似度+事实槽召回，注释明确写"不是判对错，是判'变没变'"。真正的对错判定只在 **Kimi 验收**。本地 reward 度量的是"离 V1 答案有多远"，这恰好服务于"不许漂离 V1 答案"的设计，但措辞全文统一为"漂移门"。
> - **GRPO 相对 DPO 是边际增益、量化伤精度。** 14B 实链路 humanness 0.212(SFT)→0.780(冷启动RFT)→0.836(DPO)→**0.846(GRPO)**，准确率 0.763→0.701→0.731→**0.733(守住)**。**GRPO 已跑完 100 步并完成终评（`70_grpo_report.md`：humanness 0.846、acc 0.733、verbatim_copy 7→3、explicit_ref→0），是已验证结果**；相对 DPO 是**边际增量（+0.01）**、非数量级跃升，但确已收口、不是目标值。本方案不为省显存上 QLoRA——量化会损 V1 已有准确率，与"守准"红线直接冲突。

## 一、背景与目标

### 1.1 现状

公司 V1 = 基于 Qwen2.5-32B-Instruct 微调的税务模型。它的**答案（answer）准确**，是项目要死守的核心资产。问题出在它的**推理过程（think）**：满是"根据参考问答对1""查阅检索结果"这类 RAG 机器检索腔。这是因为 V1 的 SYSTEM_PROMPT 本身就是"基于提供的参考问答对来回答……在参考问答对中搜索相关信息"的检索式提示词（见 `config.py` 的 `SYSTEM_PROMPT`），把模型逼成"看着参考抄"的口吻。对外宣称端到端，think 却暴露了检索痕迹。

### 1.2 目标与约束

- **目标**：用自我改进 RL 把 V1 的 think 优化得**像人、去 RAG 痕迹**。
- **硬约束1（守准）**：**绝不掉 V1 已有的答案准确率**。这是一票否决项。
- **硬约束2（纯自我改进）**：V1 头上没有更强的老师；**蒸馏 V1 = 已训崩的 V2 老路**。方法必须纯自我改进，不引入外部老师。

### 1.3 机器腔不是模型固有的——这是项目可行性的根据

14B 沙盒已用"裸 base 对照"实验证明：裸 base-14B 在"给参考 + 让你基于参考回答"的 prompt 下，acc=0.659、humanness=0.237（`baseline_base14b_report.md`，已核对）。这说明**机器腔主要是 prompt 逼出来的，不是模型固有能力**。因此"把 think 改自然、同时不伤 acc"在原理上成立——这是本项目敢做的根本依据。

## 二、与 14B 的关系与经验迁移

### 2.1 14B 沙盒已验证的链路与教训

- **链路**：冷启动（用 Kimi 把机器腔 think 改写成自然推导 → SFT，教模型会自然推理）→ RFT（拒绝采样自蒸馏）→ DPO（偏好对齐）→ GRPO（在线强化）。
- **教训1（必须先冷启动）**：单纯 RFT 会翻车（Goodhart：奖励只测字面，模型学会"少说关键词但语义照搬"，`verbatim_copy` 飙升）。**RL 只能放大模型已经偶尔会做的行为**——必须先冷启动给模型自然推理能力。
- **教训2（奖励要识破语义照抄）**：奖励=乘法门控并融合**条件 PMI 结构信号** `logP(think|问题+参考) − logP(think|问题+标准答案)`，相对差抵消事实难度、抗关键词规避。**注意：PMI 当前只接在 RFT 离线选样与探针，不在 GRPO 在线**（见 §6.6 与本章下方"在线照抄的兜底"）。
- **方法论沉淀**：选样用本地奖励、判阶段成败用独立金标准 Kimi、上 GPU 前先用信号探针证伪（AUC ≥ 0.7）。

### 2.2 14B 真实结果（诚实更新）

| 阶段 | humanness | acc |
|---|---|---|
| 裸 base-14B 基线 | 0.237 | 0.659 |
| SFT（冷启动后） | 0.212* | 0.763 |
| RFT | 0.780 | 0.701（回撤） |
| DPO | 0.836 | 0.731（收回） |
| **GRPO（最终）** | **0.846** | **0.733（守住）** |

> \* SFT 这一档 humanness 偏低是评测口径所致；真正的 humanness 抬升发生在 RFT/DPO/GRPO。关键事实：**准确率 SFT 0.763 → RFT 回撤 0.701 → DPO 收回 0.731 → GRPO 守住 0.733**——证明回撤可逆、守准机制有效。**GRPO 已跑完 100 步并完成终评（`70_grpo_report.md`：humanness 0.846 / acc 0.733 / verbatim_copy 7→3 / explicit_ref→0），是已验证结果**；相对 DPO 是边际增量（+0.01），非数量级跃升。

### 2.3 14B 的已知局限（32B 必须改进）

- 采样只覆盖 query 池约 2/3，且每轮**死取前 400 题**（rollout）/前 800（种子）→ 32B 改为**全量 + 随机轮换采样**。
- 训练循环是**单卡假设**（14B 跑在单张 96G 卡上）→ 32B 需多卡化（见 §4）。

### 2.4 32B 与 14B 的关键差异（必须讲清）

- **V1 就是要改的模型本身**（不是把老师蒸馏进学生）：所以**跳过蒸馏 SFT**；LoRA 的 base = V1 权重本体；直接从 V1 做 冷启动→RFT→DPO→GRPO。
- **V1 自产 think+answer**：V1 的 answer = 准确率金标准（RL 全程不能漂离），V1 的 think = 冷启动改写的原料。
- **评测口径**：student = V1+LoRA（各阶段 adapter），teacher = 原始 V1（裸权重）。

**在线照抄的兜底（替代"PMI 在线"的诚实说明）**：GRPO 在线 reward 不含 PMI，照抄由两道机制兜底：① humanness 表面项里的 `copy_signal`（LCS + 5-gram 重叠，已在 `humanness()` 内）；② **answer 段重 KL=0.1** 锚住答案不漂。`verbatim_copy` 由独立金标准与探针在留出集上周期性监控（见 §6 风险1、§7-阶段6）。

### 3.2 奖励函数（代码真实形态，已按审查修正）

本地奖励 `R = score_rollout(...)` 的真实实现（`reward.py` 第 239–247 行，已逐行核对）是**两级门控**，而非"任一门为 0 则 R=0"的描述：

```
若 格式不过(format_fail):         R = -1.0            # 负值，给 GRPO 组内强负梯度，不是 0
否则若 R_acc < τ_acc(acc_drift):  R = 0.1 · R_acc      # 答案漂移软惩罚，不是硬清零
否则(ok):                         R = R_acc · (0.5 + 0.5 · R_human)
```

其中：
- `R_acc` = **答案漂移度**（生成 answer 与 V1 answer 的 SequenceMatcher 相似度 + 事实槽召回）的连续值，**不是答对率**；`τ_acc`（`REWARD_TAU_ACC`）当前 0.30（RFT 选样硬门是 `R_acc ≥ 0.6`，亦为漂移阈而非"答对"）。
- `R_human` = humanness = 表面项 `1/(1 + c_trace·引用词数 + c_copy·照抄率)` **＋条件 PMI（32B 本轮决定打开：探针 / RFT 选样 / GRPO 在线均融合）**。PMI 的"尺子模型"是度量工具（**不是被奖励对象——奖励的始终是当前模型的 think**），由探针 AUC 选优（当前模型 / 冻结 πref / 干净 Qwen 底座，排除原始 V1）；reward 全程 **detached**（无梯度穿过 PMI），GRPO 组内归一化吸收尺子的跨步漂移。
- 注意：`R_acc` 作为**外层乘子**贯穿 `R_acc·(0.5+0.5·R_human)`，不是"0.5·humanness + 0.5·准确率"的加法平均。

> 设计语义：格式坏给 −1 是有意为之（组内归一化下提供强负梯度）；漂移软门 `0.1·R_acc` 而非清零，避免把"略漂但仍接近"的样本一棒打死；`R_acc` 当外层乘子，保证"答案越漂、humanness 加成越被压低"，从结构上锁住"不许拿准确率换像人"。

### 4.2 训练框架选型：ms-swift 当引擎、自研 reward 当大脑（✅本次拍板）

**决策**：训练用 **ms-swift（环境已装 4.0.1）当引擎**；把"方法的命根子"——`reward.py`（两级门控 + humanness 表面项 + 漂移门 + 可选 PMI）与选样/构对逻辑——**原样保留、移植成 swift 的自定义 reward 插件**。框架只换执行引擎，方法本身不动。

**为什么是 swift（不再自研单卡多卡化）**：
- 32B + 8 卡下，多卡训练（DeepSpeed ZeRO / FSDP）与 vLLM 采样 **swift 原生支持**——省掉把 14B 单卡手写循环改造成 8 卡 DDP 的重写工程（原"DDP 是重写训练循环级别"的难点由此规避）。
- 公司已用 swift（机器上有 `hkl_swift` 环境），标准化、好维护、好交接。
- swift 的 SFT/DPO/GRPO/vLLM 均经大规模验证。

**逐阶段映射**：

| 阶段 | swift 怎么跑 |
|---|---|
| 冷启动 SFT / CS_RFT | swift `sft`（多卡 DeepSpeed/FSDP）；RFT 的"采样→本地奖励打分→选样"用我们的 reward（+后期 vLLM 采样），选出的高分样本喂 swift `sft` |
| DPO | swift `rlhf --rlhf_type dpo`，喂我们用本地奖励构好的偏好对（chosen/rejected） |
| GRPO | swift `rlhf --rlhf_type grpo` + **移植 `reward.py` 成自定义 reward 插件**；rollout 用 swift 内置 vLLM |
| 评测推理 | 单卡 / 数据并行 vLLM 批量（留出集仅 224 条） |

**🔴 swift 的盲区 = 非对称 KL（必须正视）**：我们"**think 段 KL=0.02 放开 / answer 段 KL=0.1 按住**"是**按 token 分段**的；而 swift/TRL 的 GRPO 只有**单标量 KL（`beta`）**、不分 think/answer 段。三策（按优先级）：
1. **先用 swift 单 KL**，靠"**奖励漂移门 + 冷启动 + πref 锚**"守准（守准主力本就是奖励的漂移门，非对称 KL 是加的保险）；
2. 不够则 **patch swift 的 GRPO trainer**，在 KL 项上按 `</think>` 边界加 think/answer 分段权重（swift 开源、可改）；
3. 再退则**停在 DPO**（14B 实测已双赢 0.836/0.731）交差，GRPO 作冲刺、非必需。

**多卡形态（swift 下）**：32B bf16 LoRA，base 64GB < 单卡 80GB → **每卡一份完整模型 + 数据并行（DDP/ZeRO-1）**即可；真顶爆再上 ZeRO-2/3 或 FSDP（swift 一个参数切换）。**当前环境 deepspeed/vllm 暂未装**，rollout 提速 / ZeRO 分片时再单独装（避免 torch 版本咬合）。

> **重要**：原 §4.10 那批"多卡张量落点 / 去 `device_map` / 参考 adapter 多卡"的改造点，是针对**14B 自研手写循环**的；改用 swift 后**大部分由框架内部接管而失效**。真正剩下的工程 = ①把 `reward.py` 移植成 swift reward 插件；②把各阶段数据整理成 swift 的 `messages` 格式（assistant 段含完整 `<think>…</think><answer>…</answer>`）；③非对称 KL 的 swift 适配（上面三策）。**DDP=多卡训练、vLLM=高速生成，swift 各管一段、协同。**

### 4.3 vLLM 全程启用（数据构建静态服务 + GRPO 权重同步）

**本轮决定第一版就上 vLLM**（不再"先 HF 后 vLLM"）。机器上已有可用环境 **`/home/nvme02/biyh/vllm_env`（vLLM 0.20.1 / torch 2.11+cu129 / 8 卡）**，**独立于训练环境 `zhjg_rl`**（关键：别把 vLLM 装进 zhjg_rl，会改写 torch 把 swift 栈搞崩）；训练侧脚本走 HTTP/OpenAI 兼容调它。两种用法：

- **数据构建 / 评测（静态 V1）**：`vllm serve V1权重`（TP=2、port 8000）起 OpenAI 兼容服务，批量打 2239 题 / 224 留出集——**服务的是不变的 V1，起好不管**。
- **GRPO 在线（动态策略）**：rollout 用 vLLM，但服务的是**每步在变的当前策略**——所以每步 `optimizer.step()` 之后必须把**更新的 LoRA 权重（几十 MB，便宜）同步进 vLLM** 再做下一轮 rollout，保持 on-policy。**这是"边训边生成"的核心机制**：生成用的模型 = 正在训练的模型。swift GRPO 自带此权重同步，两种摆法——**colocate（生成训练同卡、原地更新）** 或 **server（vLLM 独占卡、训练在别卡、每步广播 LoRA 权重）**。
- **PMI 的冻结尺子**（πref / 干净底座）不更新 → **不在同步范围**，可常驻或单独 forward。

接入红线：vLLM 采样分布须与训练侧 logprob 前向**口径一致**（同 tokenizer / chat template / temperature / top_p），否则 advantage 与 KL 错位——上 GRPO 前用少量样本比对 vLLM 与训练前向的 logprob 一致性。

### 4.6 Kimi 改走 DashScope 公网兼容模式

**现状**：`kimi_client.py`+`config.py` 把 Kimi 指向内网 mudgate（`MUDGATE_BASE`），鉴权把同一把 key 塞进三个 header（`Authorization`/`app_id`/`appId`），且 `_session.trust_env=False`。云机连不到该 URL，所有 Kimi 判分会失败。

**具体改动清单**：
1. `config.py`：`JUDGE_BASE_URL` 改 DashScope 地址；`JUDGE_MODEL` 改占位的 DashScope 模型名；新增从环境变量读 key；停用三 header 鉴权。**并删除 `config.py` 第 36 行 `MUDGATE_API_KEY` 的明文默认值 `sk-...`**，改为 `os.environ["DASHSCOPE_API_KEY"]` 无默认（缺失即报错）——报告强调"不落明文 key"，现有代码里恰有一个明文 default，必须一并删掉。
2. `kimi_client.py`：`mudgate_headers()` 调用点改为返回 `{"Content-Type":"application/json","Authorization":"Bearer "+KEY}`；URL 拼接 `base_url+"/chat/completions"` 不变。
3. **payload 兼容性 + 判分确定性（关键修正）**：`_build_payload` 现对 Kimi 强制 `chat_template_kwargs={"enable_thinking":False}`、`do_sample=False`、`temperature=1`、`top_p=0.95`，这是 mudgate 硬约束，DashScope 可能对非标字段返回 400。**且这段是 `JUDGE_MODEL.lower().startswith("kimi")` 才触发**——若按上面把模型名改成不以 "kimi" 开头的 DashScope 名，这整段会被跳过，连 `do_sample=False`/固定温度一起没了，判分退回 `JUDGE_TEMPERATURE=0.0`/`top_p=0.7` 但不再受控。**对策：改 DashScope 后判分确定性不能依赖 `startswith("kimi")` 分支，要显式收敛到标准 OpenAI 字段（`model/messages/temperature/top_p/stream`）并显式锁定 `temperature/top_p`、移除非标字段**。

### 4.8 V1 格式适配（✅已实跑验证、原"最大隐性风险"已解除）

**结论（2026-06-08 实跑）**：用 config 的 RAG `SYSTEM_PROMPT` + 一道带参考的题、`apply_chat_template(add_generation_prompt=True)` → V1 generate，**V1 自己吐出完整 `<think>…</think>` + `<answer>…</answer>`**；think 正是"根据参考问答对…可直接引用参考问答对中的信息"的机器检索腔（项目前提在真 V1 上坐实），answer 正确。

7. **不引入量化**（仍有效）：bf16 LoRA，加载 `dtype=bfloat16`，不加 `BitsAndBytesConfig`/QLoRA。

**★ swift 化新增的核心工程项（这才是真正要做的）**：① 把 `reward.py`（两级门控+humanness+漂移门+可选PMI）移植成 **swift 自定义 reward 插件**；② 各阶段数据转 **swift `messages` 格式**（assistant 段含完整 `<think>…</think><answer>…</answer>`）；③ **非对称 KL 的 swift 适配**（§4.2 三策）。

### 4.11 上 GPU 前的工程就绪门槛（六项全绿才正式训练）

1. 环境冒烟 `torch.cuda.is_available()==True` 且 `device_count()==8`。
2. V1 本地加载 + 一条生成**不坍塌**：判据 = 生成 ≥10 条，无空输出、无复读（n-gram 重复率 < 阈值）、answer 与内网 V1 抽样一致 ≥90%。
3. Kimi DashScope 最小请求连通：返回 200 且含合法 `content`；用 3 条历史已判样本走新路径重判，分数与旧 mudgate 口径偏差 ≤ 可接受带、无 400/字段错误。
4. swift GRPO 小步冒烟（小 batch / 组大小、2 步）盯峰值：**峰值显存 ≤ 80GB×70%（即 ≤56GB）才放量**，否则下调组大小/max_len 或上 ZeRO。
5. 多卡正确性（仅当上多卡时）：**小批量多卡 loss 与单卡相对误差 < 1e-3**。
6. 信号探针在 32B 自然种子上 **AUC ≥ 0.7**（先廉价证伪、再烧卡）。

## 五、数据方案

> 一句话：32B 阶段**不重新造数据，而是把 14B 已生产、已验证的数据资产平移到 V1 上复用**；在内网检索不可达下，用"复用缓存参考"绕开在线 RAG；并修掉 14B 的两个短板（覆盖 ~2/3、死取前 400/800）升级为全量+随机轮换。

### 5.1 数据资产总览（行数已核）

| 资产 | 文件 | 规模 | 32B 角色 |
|---|---|---|---|
| 全量 query 池 | `01c_all_queries.jsonl` | 2239（usable 498 + summary 1741） | RL 池上限；V1 自产输入 |
| RL 训练/选样池 | `03_sft_train.jsonl` | 2014（已内嵌参考块 user_prompt） | 改写原料、rollout 池 |
| **固定留出验收集** | `03_sft_eval.jsonl` | **224** | 全程不训练，唯一评测口径 |
| V1 自产 think+answer | `02_company_outputs.jsonl` | 2239 | answer=金标准；think=改写原料（**32B 需用本机 V1 重产，见 5.6**） |
| 自然种子（已改写打分） | `12_seeds_scored.jsonl` | 800 | 冷启动种子来源 |

### 5.2 14B 经验/数据如何迁移复用

迁移的不是模型，是**数据资产 + 流水线 + 已调好的常量**：
- **跳过蒸馏 SFT**：`03_sft_train.jsonl` 作为 query+参考的载体保留，不再作"蒸馏目标"。
- **query 池直接复用**：2239/2014/224 原样沿用，去重（精确 + 模糊 SequenceMatcher 防"个税怎么算 vs 如何计算个税"同义泄漏）14B 已做完。
- **reward 阈值作先验但不盲信**：14B 在 224 条对齐样本 tune 出的 `REWARD_TAU_ACC=0.30 / C_TRACE=0.34 / C_COPY=1.00 / W_PMI=0.5` 作起点。**正式烧卡前用 step13 在 V1 自然种子上重新证伪（AUC≥0.7）**。分支：**AUC<0.7 → 在 V1 种子上重 tune τ/C/W（step08c 同流程）→ 仍 <0.7 则阻塞，不进 RFT**。
- **改写/打分话术复用**：Kimi 改写 prompt（保数字/税率/结论、只删检索腔）、humanness rubric 沿用，仅切公网 DashScope。

**唯一需 32B 重新生产的数据 = V1 本体自产 think+answer**（见 5.6）。

### 5.3 RAG 参考在内网不可达下的解决方案

**① 已有 query（绝大多数）——复用 14B 已缓存参考，零网络依赖（首选）。** 参考问答对早已物化在三处：`02_company_outputs.jsonl` 每行 `user_prompt`（`【参考问答对】…【问题】query`）、`03_sft_train.jsonl` 每行 `messages[1].content`、源 Excel "参考知识TOP5"列（经 `step15.parse_refs` 解析）。rollout/冷启动/选样**只读这些缓存字段**，覆盖 2239 全量池，支撑全链。
- **完整性校验门槛**：校验 2239 条中 `user_prompt` 含合法 `【参考问答对】` 块的比例 **≥ 95%**；缺失行单独登记并从训练池剔除或补取。

**② 新增 query（若后续公司给真未训过的日志）——另定参考获取。** 14B 踩过坑：原以为"一阶段输出 Excel"是新 query，实跑 step15 发现合格新 query=0（该 Excel 正是造 SFT 问题的源，`SUMMARY_XLSX==NEW_QUERY_XLSX`）。教训：**"未用过"必须实跑去重确认，不能估算**。三条路径按可行性排序：
- **(a) 离线导出 + 随包带参考（推荐）**：请内网侧用同一检索接口对新 query 批量跑，导出"query+参考TOP5"拷到本机，用 `step15` 同逻辑解析、`build_user_prompt` 拼装，与训练壳**完全一致**。
- **(b) 临时开内网路由/代理**：仅为取参考开临时通道，取完即关（云机合规/成本高，非必要不走）。
- **(c) 暂不引入新 query（默认兜底）**：沿用 14B 结论，扩 query 退化为"用满 2014 全池+随机打散"，泛化由 224 留出集证。
- **硬门**：新 query 入池前断言 `extract_references(user_prompt)` **非空**——`extract_references` 是字符串切片，壳标记（`【参考问答对】`/`【问题】`）任何偏差都会让 `copy_ratio=0` → humanness 虚高 → 照抄检测**静默失效**。

**底线**：主线（冷启动→GRPO）用 ① 即可全程跑通，不依赖内网；② 只在"要额外证泛化"时启动。

### 5.4 采样改进：全量 + 随机轮换

- **14B 短板**：`step08` 用 `samples[:max_queries]`（`RL_ROLLOUT_MAX_QUERIES=400`）、种子 `SEED_MAX=800`，确定性取前 N → 只覆盖 ~2/3、反复薅前排、尾部长 prompt（~6000 token）从未训练见过、过拟合风险。
- **32B 改法**：配额改全量（`RL_ROLLOUT_MAX_QUERIES`/`SEED_MAX` 默认 0）；GRPO 每步从全池**无放回随机抽**，走完一 epoch（2014 条）再洗牌，固定 seed（如 42）保可复现；落点把 `samples[:N]` 替换为 `random.Random(seed).shuffle` 后按步切片。
- **为何"全池随机轮换"而非"等比例切成不相交块"（本轮讨论确认）**：只有 2014 条，切 4 块每块 ~500 会饿着各阶段；且要泛化的是 **think 风格这个全局属性**，每阶段都该见广分布、query 越多越好；真正的泛化保证是 **224 留出集（永不训练）**。**各阶段同池复用不算过拟合**：RFT/DPO/GRPO 的目标是模型自生成、每阶段模型变了内容也变（自我改进、非背固定标签），answer 段被 KL 锚住、改的是通用风格；RLHF 标准做法也是各阶段共用同一 prompt 分布。
- **c/d 数据用法**：冷启动改写**覆盖全部 2014**；过门槛的高质量改写(c)入冷启动集，**低质量改写(d) 与原始机器腔 think 留作奖励探针的"低分对照"**（探针 = c 高分 vs 机器腔低分，调 reward/选 PMI 尺子到 AUC≥0.7）。
- **时长盒（排期依据）**：先用 `--smoke` 实测单步 rollout 墙钟，估算 `单步耗时 × 步数` 的时长上界；设"**单阶段 rollout 墙钟 ≤ X 小时，超时则降候选数**"，使 §8 排期有据。
- **验收**：被采样 unique query 覆盖率从 ~2/3 提到 **≥95% 全池**；记录尾部长 prompt（>3000 token）是否被采到。

### 5.5 冷启动种子：规模与质量门槛

- **来源**：Kimi 把 V1 自产机器腔 think 改写成自然推导（保全部数字/税率/结论，只删检索腔），公网 DashScope 离线跑。
- **三重质量门（全部沿用 14B）**：① `facts_ok`（改写后数字/税率/结论与原 think 一致才留，14B 实测 800 里仅 ~2% 被丢）；② **humanness ≥ 0.60**（14B 调出的关键阈值，种子重度左偏，从 0.40 抬到 0.60 只少 ~68 条却砍掉全部"将就样本"——冷启动是教风格，纯度比数量重要）；③ think 长度 `[40, 2000]` 字符。
- **规模门槛（两章已统一）**：32B 改全量改写（2014 池），过三重门后**最小 ≥ 800 条必须、目标 1300+**；不足则放宽改写轮次/重采，**不放宽事实门**。
- **冷启动 eval 特别约束（14B 踩坑修正）**：留出 eval 必须从**同分布自然种子**确定性切 ~10%（`COLDSTART_EVAL_FRAC=0.10`），**绝不能用 SFT_EVAL（V1 机器腔 think）当 eval**——否则模型越自然、机器腔 eval 上 loss 越高，早停会朝反方向选（14B 被对抗验证抓到过）。

### 5.6 V1 自产 think+answer：一份两用

- **`answer` = 准确率金标准（全程红线）**。被当作每条 query 的 `gold_answer`，贯穿：① reward 的**答案漂移门**（漂移 R_acc<τ 则软惩罚，见 §3.2，注意这是漂移度不是答对率）；② Kimi 验收时"以 V1 答案为绝对正确"判 correct/partial/incorrect（**真正的对错在这里判**）；③ GRPO 的 answer 段重 KL=0.1 锚住。
- **`think` = 冷启动改写原料**，只在冷启动这一步被消费；RFT/DPO/GRPO 的 think 是模型自采、要优化的对象。
- **为何必须用"本机这版 V1"重产、而非复用 14B 的 02 文件**：14B 的 `02_company_outputs` 是经内网在线 RAG 调 V1 拿到；32B 评测口径 teacher=原始 V1，金标准与改写原料必须出自**组长拷到本机、本地推理的这版权重**，口径才严格对齐（否则金标准与被评模型不同源，"守住没守住"无法干净论证）。这是 32B 唯一重新生产的数据（2239 条，本地批量）。

### 5.7 评测口径（三条铁律）

- **口径定义**：student=V1+LoRA（各阶段 adapter），teacher=原始 V1（裸权重）。每阶段同一留出集对打，看 ① think humanness 是否升、RAG 痕迹（explicit_ref/ref_enumeration）是否降；② answer 准确率是否守住 teacher。
- **固定留出集**：全程只在 224 条 `03_sft_eval.jsonl` 评测，从不参与训练，纵向曲线才可比。
- **选样器 ≠ 验收器（防 Goodhart）**：选样用本地 reward；判成败用独立金标准 Kimi（跨家，规避同源 self-preference）。
- **同批 Kimi 重判（抗批次飘）**：把各阶段 teacher+student 输出放进同一批 Kimi 一次性重判，统一温度/top_p（**显式锁定**，不依赖 `startswith("kimi")` 分支，见 §4.6）、同 rubric。批次一致性验收：**同一 teacher 输出在本批内重复判 2 次，humanness 差 ≤ 0.03，否则该批作废重判**（直接针对 14B 的 0.16↔0.29 飘）。
- **产出物**：每阶段 `*_infer.jsonl`、`*_judge.jsonl`、`*_report.md`（humanness 均值/分布、correct+partial 占比、痕迹计数三表对照 teacher）。

## 六、风险与对策

核心矛盾：在【无更强老师】约束下，既要 think 像人，又要一寸不让守住 V1 准确率。前两类是会"训崩"的致命风险，后四类是会"悄悄变质"的隐性风险。

### 风险 1（★最严重）：Goodhart 重演（humanness 是假的）
- **描述**：14B 已踩过的坑。奖励只测表面（引用词/照抄率），模型学会"少说关键词但语义照搬"，`verbatim_copy` 飙升。32B 容量更大，钻空子能力更强。**且在线 GRPO reward 不含 PMI（见三条诚实声明），这正是最易飙升的环节，只有表面项在守。**
- **触发信号**：`verbatim_copy` 与本地 humanness 奖励正相关或不相关（健康时应负相关）；本地奖励与金标准 Kimi 系统性背离（本地高、Kimi 低）；GRPO 中本地 reward 涨但金标准 humanness 不涨/跌。
- **对策**：① **离线选样（RFT/探针）用条件 PMI**（相对差抵消事实难度、不依赖关键词，靠规避词骗不过）；② **两级门控**（格式坏 −1、漂移 0.1·R_acc、其余 `R_acc·(0.5+0.5·R_human)`），堵"拿准换像人"；③ **选样/判分双轨分离**，本地过拟合污染不了验收；④ **先冷启动再 RL**；⑤ **在线照抄兜底**：`copy_signal`（LCS+5-gram，已在 humanness 内）+ answer 段重 KL；⑥ **GRPO 全程在留出集用独立 Kimi 周期监控 `verbatim_copy`**，定量门槛见 §7-阶段4/6。

### 风险 2（★）：答案准确率下滑（为了像人把准说没了）
- **描述**：底线，绝不能破。风险来自 think 改太自由带偏 answer，或 RL 探索推离 V1 高准区。
- **触发信号**：student acc 相对 teacher 下降趋势；GRPO answer 段 KL 持续放大。
- **对策**：① 准确率金标准锚 V1 answer；② **非对称 KL**（think 0.02 放开 / answer 0.1 按住）；③ 漂移门作乘子；④ 每阶段 Go/No-Go 卡 `acc ≥ acc0−ε`（ε 见 §7-阶段1 定死）。**14B 背书**：humanness 大涨同时 acc 0.763→0.701→**回收到 0.731**，证明回撤可逆。

### 风险 3：V1 权重/网络依赖（基础设施单点）
- **描述**：云机不在内网，链路依赖 (a) V1 权重落本机本地推理+本地 LoRA；(b) Kimi 改公网 DashScope。
- **对策**：① **V1 本地化是 M0 硬门**；② Kimi DashScope（key 存环境变量）；③ 参考优先复用缓存；④ 代码只放 pfs、模型/ckpt 放 NVMe，避免写爆 61G。

### 风险 4：数据利用率不足（14B 已知局限）
- **对策**：全量+随机轮换；留出集严格隔离；640GB 算力支撑全量（覆盖率 ≥95%，见 §5.4）。

### 风险 5：量化诱惑（QLoRA 省显存反伤 V1 精度）
- **描述**：QLoRA 量化**直接损 V1 精度**，与守准红线冲突。
- **对策**：**明确不用 QLoRA**（640GB 全空，bf16 LoRA 轻松，无省显存必要）；base=V1 全精度 bf16；把"全程不量化"写进基线 checklist。

### 风险 6：在线漂移（GRPO 训着跑偏）
- **触发信号**：训练曲线震荡/发散；answer 段 KL 异常增大；多卡 loss 与单卡不一致；组内 advantage 方差异常。
- **对策**：① πref 锚**上一阶段冻结策略**（不是原始 V1，避免把已学的自然推理拽回去）；② 组内归一化 advantage + 非对称 KL；③ **多卡先验证再放量**（单卡=多卡 loss 误差 <1e-3，且第一版优先单卡，见 §4.2）；④ 上 GPU 前探针 AUC≥0.7；⑤ 定期金标准体检，掉头即回退上一冻结 ckpt。

## 七、逐阶段实施（每阶段：需要什么 / 验收什么 / 落地什么）

> 全局规则（修审查共性硬伤）：
> - **每阶段入口先做上游产出物校验**：上一阶段 checkpoint 已落 nvme 指定路径、可被本阶段成功 load（冒烟 load + 1 步前向），否则不开工，堵静默用错权重。
> - **回退/降级规则**：任一阶段 No-Go → 回退到上一冻结 ckpt 排障；**连续 2 次 No-Go 触发评审**：要么放宽**非红线**软指标（humanness 目标），要么以上一稳定阶段产物降级交付；**红线 ε（守准）永不放宽**。
> - **M0 期间可并行无依赖项**（环境、Kimi 改造、采样代码改造、reward 实现、V1 格式适配脚本可在权重到位前先做），把关键路径从全串行压缩。

### 阶段 0：准备（环境 / 权重 / 数据通路 / V1 格式适配）
- **需要**：8×A800；V1 权重落本地 NVMe（严禁 pfs）；conda（`hkl_py310_3` 或新建 cu121，禁 cu130）；公网 Kimi DashScope（key 存环境变量）；14B 缓存参考。
- **验收（量化门槛）**：
  - 8 卡全可见且 `cuda=True`；bf16 LoRA 前向反向 smoke 通过。
  - V1 本地推理产出 (think,answer)，answer 与内网 V1 抽样一致（人工抽查 ≥10 条无明显漂移）；生成不坍塌（无空输出/复读 < 阈值）。
  - Kimi DashScope smoke 返回 200 含合法 content；记录单次 token 用量，估算全链路（2239×N 次重判）成本上界，超预算报警。
  - **参考缓存命中率 ≥ 90% 训练池**方可进 M1；低于此先走纯缓存子集并在阶段1记录子集规模。
  - **V1 格式适配确认**（§4.8）：V1 是否自动注入开 `<think>`、`parse_think_answer`/`kl_weights` 边界是否对齐、SYSTEM_PROMPT 处置方案已定。
- **落地**：可复现环境清单（env 名 + 锁版本）；**V1 本地推理脚本（新增产出物，非改造）** + Kimi DashScope 判分脚本；数据通路盘点表；目录磁盘约定文档；V1 格式适配结论与改点记录。

### 阶段 1：V1 基线评测（锚定 teacher 基线 + 定死红线常量）
- **需要**：阶段0的 V1 本地推理 + Kimi 判分；固定留出集 224 条（全程冻结）；基线对照（原始 V1，可选裸 base-32B）。
- **验收（量化门槛）**：
  - 产出 V1 在留出集 (acc0, humanness0)，可复现（同集同脚本两次跑分差异 < 1%）。
  - 机器腔被量化：引用词平均频次、照抄率（`verbatim_base`）数值。
  - **显式产出并写进 config 三个常量**：`acc0=<实测>`、`humanness0=<实测>`、**`ε=<定值>`（建议 ε=0.03，口径"acc ≥ acc0−ε"）**、`verbatim_base=<实测>`。**ε 必须在这里定死**——否则后续 6 个阶段"acc≥acc0−ε"全部不可判定（这是全方案最致命的开放项）。**32B 口径注意：student 起点=V1 本体，故 acc0≈1.0（V1 跟自己比）**，ε 实际守的是"别漂离 V1 原答案超过 ε"；阶段1 真正测得的关键量是 humanness0（机器腔基线）。
- **落地**：基线评测报告（acc0/humanness0/引用词频次/照抄率）；冻结的 hold-out 评测集；标准化评测脚本（一键复现，各阶段复用）；红线常量定值（写进 config）。

### 阶段 2：冷启动（CS，给自然推理种子）
- **需要**：V1 自产 (think,answer)；公网 Kimi 改写器；facts_ok 校验 + humanness 打分器。
- **验收（量化门槛）**：
  - 入选样本 `facts_ok=100%`、`humanness≥0.6`、think 长度 `[40,2000]`。
  - 过双筛入选 **≥800 条（目标 1300+）** 方可 SFT；不足放宽改写轮次/重采，**不放宽事实门**。
  - CS 模型（V1+LoRA）留出集：**humanness ≥ humanness0 + 0.15**（绝对方向达标），且 **acc ≥ acc0−ε**（后续每阶段复用的硬门）。
  - query 覆盖率达全量（不再 2/3）。
  - 冷启动 eval 用同分布自然种子切 10%，**不得用 SFT_EVAL**。
- **落地**：冷启动改写数据集（原 think/改写 think/facts_ok/humanness）；CS 阶段 LoRA adapter（存 nvme）；CS vs V1 (acc,humanness) 对比表；改写 prompt 模板 + 阈值记录（含"为何保数字删检索腔"决策）。

### 阶段 3：奖励校准与证伪（先证伪再烧卡）
- **需要**：带标签探针样本（人类自然 think vs 机器腔/语义照抄 think）；奖励各分量计算器（格式门/漂移门/humanness 表面项/离线 PMI）；固定留出集（与选样器分开）。
- **验收（量化门槛）**：
  - 奖励区分力 **AUC ≥ 0.7**（独立留出集）。**本轮新增：PMI 打开，并在 c(高质改写) vs 机器腔(低) 上用探针对比候选"尺子模型"（当前模型 / 冻结 πref / 干净 Qwen 底座），AUC 最高者即 PMI 尺子，排除原始 V1。**
  - 反 Goodhart 专项：构造"少引用词但语义照抄"对抗样本，**被判低分比例 ≥ 90%**（PMI 项揪出）。
  - **门控正确性（按代码真实形态，修正旧错误验收标准）**：格式坏样本 `R=−1.0`、漂移样本（R_acc<τ）`R=0.1·R_acc`、其余 `R=R_acc·(0.5+0.5·R_human)`，单测全过。（**旧草稿"答错 R 必为 0/格式坏 R 必为 0"是错误标准，会让单测必然失败**。）
- **落地**：奖励函数实现（各分量 + 两级门控合成）；探针证伪报告（AUC + 对抗样本表现 + 阈值/权重定档）；校准好的奖励配置；决策日志（为何相对 PMI 差、为何门控这样设、**为何 PMI 只接离线**）。

### 阶段 4：冷启动 RFT（拒绝采样自蒸馏）
- **需要**：阶段2 CS 模型；阶段3 校准的本地奖励（**离线选样含 PMI**）+ gate；全量+随机轮换采样器。
- **验收（量化门槛）**：
  - CS_RFT 留出集：humanness 较 CS 进一步上升或持平且**更稳（留出集多次评测 humanness std ≤ 0.03）**，**acc ≥ acc0−ε**。
  - **`verbatim_copy ≤ CS 阶段值，且绝对值 ≤ 0.05；若较 CS 上升 > 0.02 → No-Go`**（反 Goodhart 关键观测；14B 正是这里翻车）。
  - 入池样本 100% 过 gate（`R_acc ≥ 0.6` 漂移硬门 + 格式正确）。
- **落地**：拒绝采样自蒸馏数据集（候选+本地奖励分+入池标记）；CS_RFT adapter（存 nvme）；留出集对比（vs CS、vs V1，含 verbatim_copy 趋势）；rollout/采样配置（温度/候选数/轮换策略，**第一版单卡**）。

### 阶段 5：DPO（偏好对齐）
- **需要**：CS_RFT 模型 + **CS_RFT 冻结副本作 πref（预计算 logP）**；候选池 + 本地奖励 R 与 humanness 分项 R_human。**偏好对来自 CS_RFT 的新 rollout（vLLM 重采样，比复用 RFT 旧 rollout 更 on-distribution）**。
- **2026-06-11 实现纠错**：旧 LoRA 脚本漏传 `--ref_adapters`，实际 πref 会退回原始 V1。修正主线先将 V1+最终 RFT LoRA 合并成完整 RFT base，再在该 base 上新建 DPO LoRA；禁用新 LoRA 即严格得到冻结 RFT πref，消除双 adapter 歧义。
- **验收（量化门槛）**：
  - 偏好对：chosen=reward 最高 / rejected=R_human 最低，**二者均 gate-ok 100%**，且 **R_human 差 ≥ margin（取 0.2）100%**（不达标的对剔除）。
  - DPO 留出集：**humanness 较 CS_RFT ≥ +0.05**，**acc ≥ acc0−ε**；引用词频次/照抄率持续下降。
- **落地**：DPO 偏好对数据集（chosen/rejected + R + R_human + margin）；DPO adapter（存 nvme）；πref 预计算 logP 缓存；留出集对比（vs CS_RFT）+ 机器腔趋势图。

### 阶段 6：GRPO（在线强化）
- **需要**：DPO 模型作策略起点；**上一阶段冻结策略作 πref**（锚点，非原始 V1）；在线本地奖励（**表面项，不含 PMI**）+ 全量随机轮换；**第一版单卡**（多卡/vLLM 作增量）；上卡前探针 AUC≥0.7。
- **2026-06-11 实现纠错**：DPO LoRA 先合并为完整 DPO base；GRPO 分别从完整 RFT base / 完整 DPO base 新建 LoRA。每条 GRPO 禁用自己的新 LoRA 后均严格回到对应冻结基座，不再依赖隐式 `ref_adapters`。
- **做法**：组内采样多候选 → 本地奖励 → **组内归一化得 advantage**（无 critic，契合无更强老师约束）；**非对称 KL（think 0.02 放开 / answer 0.1 按住）**——改 think 不掉 acc 的核心。**本轮决定**：① **在线 reward 打开 PMI**（尺子=探针选优的固定 πref / 干净底座，不用每步在变的活策略；reward detached + GRPO 组内归一使其稳）；② **rollout 用 vLLM（vllm_env），每步 `optimizer.step()` 后把更新的 LoRA 权重同步进 vLLM 保 on-policy**（swift colocate/server 模式，见 §4.3）。
- **验收（量化门槛）**：
  - 终态 humanness 较 DPO 有提升（**GRPO 对 DPO 是边际增量，不强求数量级跃升**；14B 实测 0.836→0.846、+0.01，acc 0.731→0.733 守住，已收口）。
  - **acc ≥ acc0−ε 全程不破**；answer 段 KL=0.1 起按住作用。
  - 训练稳定：**KL 移动均值 ≤ 上限（think 0.05 / answer 0.2）**，reward 不出现连续 K step 单调钻空（配合 verbatim 监控），`verbatim_copy` 不反弹（守 §7-阶段4 的 ≤0.05/涨幅≤0.02）。
  - **在线早停**：留出集抽测 acc 跌破 `acc0−ε` 立即早停/回退上一冻结 ckpt。
- **落地**：GRPO adapter（最终交付主体，存 nvme）；在线训练曲线（reward/KL/acc/humanness vs step）；训练脚本骨架与超参（组内候选数、think/answer KL、归一化方式）；早停/回退记录与决策日志。

### 阶段 7：终评与交付
- **需要**：各阶段 adapter（CS/CS_RFT/DPO/GRPO）；固定留出集 + 独立金标准 Kimi（与选样器分开）；原始 V1 并排对照。
- **验收（成功判据，硬数字）**：
  - **像人**：最终 student humanness **≥ 0.70 且 ≥ 3×humanness0**（硬线，"参照 4×"不是门槛）。
  - **守准（一票否决）**：最终 **acc ≥ acc0−ε**。
  - **机器腔基本消除**：引用词频次、verbatim 照抄率降到目标线（`verbatim_copy ≤ 0.05`）。
  - **两套裁判判同向（无 Goodhart 背离）**：本地 reward 排序与 Kimi 排序 **Spearman ρ ≥ 0.6**，且无某阶段"本地↑而 Kimi↓"。
  - **可复现**：交付脚本+配置在本机重跑出同档结果（容差内）。
- **落地**：终评报告（V1 vs 最终 student 并排：acc/humanness/机器腔全表 + 演化曲线）；最终交付模型 V1+GRPO LoRA（含各中间 adapter，存 nvme）；全链路复现包（环境清单/各阶段脚本骨架/奖励配置/评测脚本/留出集）；经验沉淀文档（14B→32B 迁移结论、决策心路历程 why/弯路/推导）；上线建议（是否作新 V1 think 引擎、剩余风险：新增 query 参考获取、内网集成、PMI 是否上在线）。

## 九、里程碑与排期（粗粒度相对周，含 Go/No-Go）

排期受"V1 权重到位时间"与"参考数据齐备度"两个前置牵引。**No-Go 即不进下一阶段**（带病推进只会越走越偏）；每个 M 的 Go 增加"产出 ckpt 已落 nvme 指定路径、可被下一阶段 load（冒烟 load+1 步前向）"。

| 里程碑 | 周 | 需要 / 验收产出 | Go / No-Go 门槛 |
|---|---|---|---|
| **M0** 基础设施 + V1 本地化 + 格式适配 | W1 | 权重落 nvme；环境就绪；DashScope 通；缓存参考；V1 格式适配确认 | **进行中**：✅V1 已解压确认 Qwen2.5-32B / ✅环境 `zhjg_rl`(swift 4.0.1, 8卡) / ✅格式实跑验证(自吐 `<think></think><answer></answer>`、风险解除) / ✅Kimi DashScope 连通；⏳ 待：代码+数据上传 `/mnt/pfs/zhjg`、Kimi 判分接进代码、写 V1 批量推理脚本、确认 Kimi model id。⛔ 余项未完不进 M1 |
| **M1** 基线评测 + 定死红线常量 | W2 | 实测并写入 config：acc0、humanness0、**ε（0.03）**、verbatim_base | ✅ 两次跑分差异<1% 且常量已定死写进 config；⛔ ε 未定 → 整套 Go/No-Go 不成立，不进 M2 |
| **M2** 冷启动 SFT | W2–W3 | facts_ok=100%+humanness≥0.6，入选≥800 | ✅ humanness ≥ humanness0+0.15 且 acc≥acc0−ε；⛔ 事实校验大面积不过/acc 掉线 → 回炉改写 |
| **M3** RFT 拒绝采样 | W4 | CS_RFT 冻结策略 | ✅ humanness 升+std≤0.03 且 verbatim_copy≤0.05 且涨幅≤0.02 且 acc≥acc0−ε；⛔ verbatim 飙升/acc 滑 → 回查奖励/补 PMI 选样 |
| **M4** DPO | W5 | πref=CS_RFT 冻结预计算；偏好对 margin≥0.2 | ✅ humanness 较 M3 ≥+0.05 且 acc≥acc0−ε；⛔ acc 破线/无增益 → 调 margin/偏好对构造 |
| **M5** GRPO | W6–W8 | πref 锚上阶段；非对称 KL（0.02/0.1）；第一版单卡；上卡前 AUC≥0.7 | ✅ humanness 较 DPO 有增量 + acc≥acc0−ε + KL 均值≤(0.05/0.2) + 稳定不发散；⛔ 漂移/发散/answer KL 失控/acc 破线 → 回退上一冻结 ckpt；探针 AUC<0.7 不烧卡 |
| **M6** 总体验收与交付 | W9 | 全链冻结 ckpt + 留出集 + 独立金标准 + EXPERIMENT_LOG | ✅ §八四条判据全满足（humanness≥0.70 且≥3×humanness0 / acc≥acc0−ε / 纯自我改进 / 可复现）；⛔ 任一不满足 → 定位失守环节回退修复 |

> **排期总览**：M0(W1)→M1(W2)→M2(W2–W3)→M3(W4)→M4(W5)→M5(W6–W8)→M6(W9)，约 9 周。M0 的 V1 本地化与参考齐备是全局前置，延迟则整体顺延（但环境/Kimi/采样/reward/格式适配可在 M0 期间并行先做）。M5（GRPO）算力与工程难度最集中，预留最长窗口（3 周）。**全局降级规则**：任一里程碑连续 2 次 No-Go → 评审决定放宽非红线软指标或以上一稳定阶段产物降级交付；**红线 ε 永不放宽**。
